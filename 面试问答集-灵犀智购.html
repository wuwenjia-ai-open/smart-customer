项目速览

技术栈：LangGraph · LangChain · Neo4j · FastAPI · Vue 3 · Ollama bge-m3 · MySQL · Docker

架构：四分支主图（闲聊/拒答/知识库/识图）+ 六节点子图（审核→规划→工具选择→并行执行→总结→幻觉校验）

数据规模：Neo4j 知识图谱 50+ 节点（产品/订单/评价/FAQ/售后），26 条预定义 Cypher + Text2Cypher 动态查询

第一类：架构设计类问题（15题）

Q1：你的项目为什么选择 LangGraph 而不是直接调 LLM API？

Situation：客服场景涉及多步骤推理——用户问"帮我挑一个性价比最高的智能音箱"，需要先查产品列表、再查评价评分、最后综合推荐。单次 LLM 调用无法完成这种链式任务。

Task：设计一个能处理多步骤推理、有状态记忆、可中途纠正的 Agent 架构。

Action：

1. 选型对比——评估了三种方案：（a）纯 LLM + Prompt Engineering（无状态，多步推理不可靠）；（b）LangChain AgentExecutor（黑盒循环，不可控）；（c）LangGraph StateGraph（显式状态图，完全可控）

2. 选择 LangGraph 核心理由——显式状态管理（StateGraph + TypedDict）、条件路由（conditional_edges）、并行分发（Send()）、内置持久化（MemorySaver + SQLite checkpoint）、可观测（每一步的状态变化都可追踪）

3. 架构设计——主图 4 分支路由（路由→闲聊/拒答/知识库/识图），子图 6 节点 Plan-and-Execute（安全审核→任务规划→工具选择→并行执行→总结→幻觉校验循环，最多重试 3 次）

4. 对话记忆——MemorySaver 将每个 thread_id 的状态持久化到 SQLite，实现多轮对话上下文保持

Result：多步骤查询（查产品+评价+推荐）可在单次请求内完成；多轮对话记忆正确保持上下文；整个 Agent 流程完全可追踪调试。

Q2：为什么用 Plan-and-Execute 而不是 ReAct？

Situation：用户查询通常是"查XX产品的评价""有哪些智能灯泡"这类独立、可并行执行的任务。ReAct 的"想一步做一步"反而增加延迟。

Task：选择最适合电商客服场景的 Agent 范式。

Action：

1. 对比分析——ReAct（Think→Act→Observe 循环）适合需要根据上一步结果动态调整的场景（如代码调试）；Plan-and-Execute 适合任务可预先拆分的场景

2. 业务匹配——客服查询通常是确定性的（查产品、查订单、查评价），不需要中间观察再调整。先用 Planner 拆解为 1-3 个子任务（如"查智能音箱列表+查评价排名"），再通过 Send() 并行分发执行

3. 实现——Planner 节点用 LLM bind_tools([SubTask]) 拆解任务 → map_reduce_planner_to_tool_selection 返回 [Send("tool_selection", {...}) for t in tasks]，LangGraph 自动并行执行

4. 兜底——简单问题 Plan 出 1 个子任务，等价于直接执行；复杂问题最多 3 个子任务并行，避免过度拆分

Result：并行执行比串行快约 2 倍（两个子任务同时查 vs 逐个查）；对于简单问题（1 个子任务），没有额外开销。

Q3：你为什么不直接用向量数据库做语义检索，而是用 Neo4j 知识图谱？

Situation：电商数据天然是图结构——产品属于类别、订单包含产品、客户下单、评价关联产品。用向量库存储会丢失关系信息。

Task：设计能同时处理精确查询（"订单 ORD-001"）和关系查询（"买了智能音箱的客户还买了什么"）的检索系统。

Action：

1. 数据建模——Neo4j 9 类节点（Product/Category/Order/Customer/Review/ReturnOrder/FAQ/AfterSalesPolicy/Employee）+ 关系（BELONGS_TO/CONTAINS/PLACED/ABOUT/ORDERED_BY）

2. 双通道查询——预定义 Cypher（26 条，Ollama bge-m3 向量语义匹配，毫秒级）+ Text2Cypher 动态生成（LLM 写 Cypher → EXPLAIN 校验语法 → 执行 → 失败重试 1 次）

3. 工具选择——tool_selection 节点用 LLM 判断走预定义还是动态生成，优先预定义（快、安全）

4. 参数填充——预定义查询的 $parameter 由上游 LLM 从用户问题中提取

Result：关系查询（如"某类别下评价最高的 TOP 5 产品"）一条 Cypher 完成，等价于 SQL 3-4 次 JOIN；知识图谱可扩展——新增节点类型不影响已有查询。

Q4：主图和子图分别做了什么？为什么这样划分？

Situation：客服系统需要先判断问题类型（闲聊/业务/图片），再对业务问题执行知识库查询。两类逻辑职责不同，耦合在一起会难以维护。

Task：设计清晰的分层架构。

Action：

1. 主图（lg_builder.py, 5 节点）——analyze_and_route_query（路由）→ 条件分发到 respond_to_general_query / get_additional_info / create_research_plan / create_image_query。职责：意图识别 + 分流

2. 子图（multi_tool.py, 6 节点）——create_research_plan 内部嵌套。职责：安全审核→任务规划→工具选择→{预定义Cypher, Text2Cypher}并行→总结→幻觉检测

3. 划分原因——主图负责"去哪里"，子图负责"怎么查"。子图可以独立测试、独立修改，不影响主图路由逻辑

4. 状态隔离——子图有独立的 OverallState（question/messages/tasks/cyphers/summary/answer/steps），通过 OutputState 只暴露 answer/question/steps/cyphers 给父图

Result：两个图职责清晰，各自可独立调试。如果未来要加"联网搜索"分支，只需在主图加一个节点，不改子图。

Q5：langgraph 的 Send() 并行分发是怎么工作的？和普通并发有什么区别？

Situation：Planner 拆解出 2-3 个子任务后，需要并行查询 Neo4j 以提高速度。

Task：实现子任务级别的并行执行，而非 Python 层面的 asyncio.gather。

Action：

1. LangGraph 的 Send() 机制——map_reduce_planner_to_tool_selection 函数返回 [Send("tool_selection", {...}), Send("tool_selection", {...})]，LangGraph 自动将每个 Send 作为独立执行路径并行处理

2. 与普通并发的区别——Send() 是图级别的并行（每个 Send 走完整的 tool_selection→predefined/cypher→summarize 路径），结果通过 Annotated[List, add] 自动汇聚到 state.cyphers；asyncio.gather 只是函数级别的并发

3. reducer 汇聚——OverallState 中 cyphers 字段使用 Annotated[List[CypherOutputState], add]，LangGraph 自动将多个并行分支的输出合并

4. 容错——单个子任务失败不影响其他子任务，失败的 cypher 会记录 errors 字段

Result：2 个子任务并行执行时间 ≈ 单个子任务时间（而非 2 倍）；并行结果自动汇聚，不需要手动合并。

Q6：为什么需要 Guardrails 安全护栏？怎么实现的？

Situation：客服系统只服务智能家居产品。用户问"推荐一件晚礼服"时，系统必须拒绝，不能去知识库瞎查。

Task：在查询执行前拦截超出业务范围的请求。

Action：

1. 两层 Guardrails——主图路由层（Router 将非智能家居问题归类为 additional-query → get_additional_info 节点判断拒绝）+ 子图审核层（guardrails 节点用 LLM 判断 decision: "continue"/"end"）

2. 判断依据——注入业务范围描述（智能家居六大类）+ Neo4j 数据库标签（CALL db.labels()），LLM 基于此判断相关性

3. 拒绝策略——decision="end" → 直接返回 "抱歉，我们专注于智能家居产品"；decision="continue" → 继续 Planner

4. 宁可误收不误拒——prompt 中明确 "如有疑问，假设问题可能相关"

Result：非智能家居问题 100% 在查询执行前拦截，不会浪费后续 LLM 调用和 Neo4j 查询成本。

Q7：你的对话记忆（多轮对话）是怎么实现的？

Situation：用户问"有哪些智能音箱"→ 系统回复列表 → 用户追问"第一个的续航怎么样"。系统必须知道"第一个"指的是刚才列表中排名第一的产品。

Task：实现跨轮次的对话上下文保持。

Action：

1. LangGraph MemorySaver——编译图时传入 checkpointer=MemorySaver()，使用 SQLite 持久化（langgraph-checkpoint-sqlite）

2. thread_id 机制——每个前端会话对应一个 thread_id（前端用 conversationId），同一 thread_id 的请求共享 state.messages 历史

3. 消息追加——每次请求用 InputState(messages=...) 传入新消息，LangGraph 自动追加到历史 messages（Annotated[List[AnyMessage], add_messages]）

4. 前端配合——useChat.js 在 API 请求头中传 X-Conversation-ID，后端从请求中提取 thread_id

Result：多轮追问对话记忆准确；切换会话创建新 thread_id，不会混淆。

Q8：tool_selection 节点是怎么决定用预定义查询还是 Text2Cypher 的？

Situation：系统有两个查询工具——预定义 Cypher（26 条，快但固定）和 Text2Cypher（动态生成，灵活但慢）。需要根据问题复杂度选择。

Task：设计智能的工具选择逻辑。

Action：

1. 双工具注册——tool_schemas = [predefined_cypher, cypher_query]，两个都是 Pydantic BaseModel

2. LLM 选择——tool_selection 节点用 llm.bind_tools(tool_schemas) 让 LLM 根据子任务内容选择最合适的工具

3. 路由——选择结果通过 Command(goto=Send("predefined_cypher", ...)) 或 Command(goto=Send("cypher_query", ...)) 分发到对应节点

4. 预定义匹配——Ollama bge-m3 将用户问题向量化，与 26 条查询的向量做余弦相似度匹配，top-5 中过滤掉参数不匹配的，取第一条可执行的

Result：简单查询（"有哪些智能音箱"）命中预定义，毫秒级响应；复杂查询（"200元以内评分最高的智能灯泡"）走 Text2Cypher 动态生成。

Q9：Text2Cypher 怎么保证生成的 Cypher 语句正确且安全？

Situation：LLM 生成的 Cypher 可能有语法错误、逻辑错误，甚至有注入风险。

Task：确保动态生成的 Cypher 可执行且安全。

Action：

1. Few-shot 示例——prompt 中包含正确的 Cypher 示例，引导 LLM 生成规范语句

2. EXPLAIN 校验——生成后先用 EXPLAIN {cypher} 在 Neo4j 上做语法校验，不实际执行

3. 失败重试——校验失败则将错误信息反馈给 LLM，重新生成（最多重试 1 次）

4. 参数化查询——所有用户输入通过 $parameter 传参，不拼接字符串，防止 Cypher 注入

5. 只读限制——prompt 中约束只生成 MATCH/RETURN 语句，禁止 CREATE/DELETE/MERGE

Result：Cypher 语法错误率 <5%（EXPLAIN 校验 + 重试后），无注入风险。

Q10：幻觉检测节点是怎么工作的？有必要吗？

Situation：LLM 偶尔会编造不在数据库中的信息（如虚构产品名称、价格），这在客服场景是致命的。

Task：检测并纠正 LLM 回复中的幻觉。

Action：

1. 检测机制——总结节点生成回复后，check_hallucinations 节点将原始数据库查询结果 + 生成回复一起发给 LLM，让它判断回复是否完全基于数据库事实

2. 输出——decision: "真实"或"包含幻觉"，包含幻觉则返回 "summarize" 触发重写

3. 循环——hallucination_conditional_edge 判断 next_action，不等于 "end" 则回到 summarize 重新生成（最多 3 次）

4. 代价——每次检测多一次 LLM 调用，约增加 1-2 秒延迟

Result：幻觉生成被有效过滤。对于大多数查询（数据库结果清晰），1 次生成即可通过检测。

Q12：为什么选择 DeepSeek 而不是 GPT-4 或文心一言？

Situation：需要选择性价比最高的大模型 API。

Task：在效果和成本之间做出平衡。

Action：

1. 成本——DeepSeek-chat 价格约为 GPT-4 的 1/20，对于需要多次 LLM 调用的 Agent（每次请求可能调用 5-8 次），成本差距巨大

2. 效果——在中文电商场景下，DeepSeek-chat 的结构化输出和指令跟随能力足够胜任路由/审核/规划/总结等任务

3. 兼容性——通过 langchain-deepseek 包无缝集成 LangChain，支持 bind_tools 和 with_structured_output

Result：单次完整 Agent 请求（含 5-8 次 LLM 调用）成本约 0.01 元人民币，远低于 GPT-4。

Q13：Ollama + bge-m3 在项目里到底做什么？没有它行不行？

Situation：系统有 26 条预定义 Cypher 查询，需要根据用户问题匹配最相关的查询。

Task：实现快速、零成本的语义匹配。

Action：

1. 匹配流程——启动时将所有预定义查询的描述文本（名称+说明）通过 bge-m3 向量化存储在内存中 → 用户问题到来时同样向量化 → 余弦相似度对比 → 返回 top-5 最匹配的查询

2. 没有它怎么办——需要额外一次 LLM 调用来从 26 条中选（增加延迟+成本）；或直接用关键词匹配（准确率低）

3. 为什么本地部署——bge-m3 是一个小型化模型（~2GB），CPU 即可运行，不需要 GPU；本地部署零延迟、零 API 费用

4. 参数过滤——匹配后检查 Cypher 的 $参数 是否已从用户问题中提取，缺失参数的跳过，取第一条参数齐全的

Result：预定义查询匹配耗时约 200ms（含 embedding 计算），比 LLM 选择快 5-10 倍；系统启动时需要 ~10 秒初始化向量（26条 × batch_size=2 × ~400ms/batch）。

Q14：主路由（Router）是怎么判断用户问题类型的？

Situation：用户问题可能是闲聊、查产品、传图片、超范围请求，需要准确分类后走不同分支。

Task：设计四分类路由逻辑。

Action：

1. 结构化输出——model.with_structured_output(Router) 强制 LLM 返回 {type: "general-query"|"additional-query"|"graphrag-query"|"image-query", logic: "分类理由"}

2. 路由规则——general-query（闲聊/无关）→ 直接回复；additional-query（超范围/缺信息）→ Guardrails 审核；graphrag-query（查知识库）→ 子图；image-query（图片）→ 视觉模型

3. 分类提示——ROUTER_SYSTEM_PROMPT 详细描述了每类的边界（如"服装食品体育用品 → additional-query"、"产品/订单/售后查询 → graphrag-query"）

4. 兜底——未知类型 raise ValueError，不允许静默失败

Result：四分类准确，非智能家居请求正确路由到 additional-query → Guardrails 拒绝。

Q15：如果让你重新设计这个系统的架构，你会怎么改进？

Situation：当前架构是单次请求内同步执行所有节点，存在改进空间。

Task：识别架构瓶颈并提出改进方案。

Action：

1. 当前瓶颈——子图每次执行 5-8 次 LLM 调用（guardrails + planner + tool_selection × N + summarize + hallucination），首次请求约 15-22 秒；Ollama 初始化慢（26 条向量 batch 计算）

2. 改进方向一——Planner 简化：简单查询跳过 Plan，直接 tool_selection（省 1 次 LLM 调用）

3. 改进方向二——缓存：常见查询缓存 Cypher 结果（Redis），相同问题直接返回（省全部 LLM 调用）

4. 改进方向三——流式返回：summarize 结果通过 SSE 逐 token 返回，用户感知延迟大幅降低

5. 改进方向四——模型分层：简单任务用轻量模型（如 deepseek-v4-flash），复杂任务用 deepseek-chat

Result：理论上可将首次响应时间降到 5-8 秒（缓存命中时 <1 秒）。

第二类：技术实现类问题（10题）

Q16：你是如何处理 LangGraph 子图的状态传递的？

Situation：子图有独立的 OverallState，父图需要在调用子图后获取结果。

Task：正确传递和提取子图状态。

Action：

1. 输入——构造子图 input_state：{"messages": state.messages, "question": ..., "steps": []}

2. 执行——response = await multi_tool_workflow.ainvoke(input_state)

3. 输出——子图 OutputState 定义了暴露字段：answer/question/steps/cyphers。父图只取 response.get("answer", "")

4. 消息隔离——子图内部消息不污染父图（子图的 messages 不出现在 OutputState 中），避免重复显示

5. reducer——cyphers 使用 Annotated[List, add]，并行分支结果自动合并

Result：父子图状态传递清晰，子图内部消息不会泄露到主对话中。

Q17：流式输出（SSE）是怎么实现的？

Situation：Agent 回复可能需要 15-20 秒，如果用户看到空白等待会以为系统卡死。

Task：实现"对方正在输入..."状态推送 + 最终回复流式显示。

Action：

1. SSE 端点——FastAPI /api/langgraph/chat 使用 StreamingResponse，media_type="text/event-stream"

2. 状态推送——在 graph.ainvoke() 之前先 yield {"status": "thinking", "msg": "对方正在输入..."}，前端显示加载动画

3. 最终回复——ainvoke 完成后 yield {"status": "done", "answer": answer, "conversation_id": thread_id}

4. 前端解析——useChat.js 使用 fetch + ReadableStream 逐行解析 SSE，reactive() 更新 Vue 组件

5. multi-turn——thread_id 通过 X-Conversation-ID header 传递，保持对话连续性

Result：用户消息发出后立即看到"对方正在输入..."，感知等待时间大幅缩短。

Q18：知识图谱中的预定义 Cypher 是怎么匹配用户问题的？

Situation：26 条预定义 Cypher 覆盖了不同查询场景，需要快速匹配。

Task：实现精确的语义匹配。

Action：

1. 向量化——启动时用 Ollama bge-m3 将每条查询的"名称 + 描述"向量化（f"{name} {desc}"），存在内存中

2. 匹配——用户问题同样向量化 → 计算余弦相似度 → top-5 排序

3. 参数过滤——每条 Cypher 用正则 \$(\w+) 提取所需参数，检查用户是否提供了这些参数（从 state 的 query_parameters 获取），缺失的跳过

4. 执行——取第一条参数齐全且相似度 ≥ 阈值的执行

5. fallback——所有预定义都不匹配时，返回errors由 summarize 节点兜底

Result：正确匹配率约 85-90%，匹配失败的走 Text2Cypher 动态生成。

Q19：你用的 pydantic 结构化输出有什么坑吗？

Situation：LLM 需要返回固定格式的数据（路由分类、Guardrails 决策、子任务列表等）。

Task：确保 LLM 输出符合预期格式。

Action：

1. 使用的结构化输出——with_structured_output(Router) → {type, logic}；with_structured_output(AdditionalGuardrailsOutput) → {decision}；bind_tools([SubTask]) → SubTask 列表

Q20：为什么要用 SQLite 做 checkpoint 而不是 Redis？

Situation：LangGraph MemorySaver 需要持久化存储对话状态。

Task：选择适合当前规模的持久化方案。

Action：

1. SQLite checkpoint——langgraph-checkpoint-sqlite 包，零配置，单文件存储，适合单机部署

2. 不选 Redis 的原因——本地开发增加一个服务依赖；当前项目没有分布式部署需求（单实例足够）；SQLite 的读写性能对于对话 checkpoint（每次请求读一次、写一次）完全够用

3. 未来扩展——如果需要多实例部署，可切换到 PostgreSQL checkpoint（langgraph 官方支持）

Result：SQLite 方案零运维成本，对话记忆功能正常。

Q21：前端 Vue 3 的响应式有什么坑？

Q26：为什么第一次请求比较慢？怎么优化？

Situation：系统启动后第一次请求约 15-22 秒。

Task：分析瓶颈并提出优化方案。

Action：

1. 瓶颈分析——Ollama 向量初始化约 10 秒（26 条查询 × batch_size=2 × ~400ms/batch ≈ 5.2 秒，加上连接建立）；子图 5-8 次 LLM 调用，每次约 1-3 秒；Neo4j 查询约 0.2-1 秒

2. 优化一——Ollama 向量预热：应用启动时完成 VectorQueryMatcher 初始化，首次请求不需要等

3. 优化二——Planner 短路：简单问题（1 个 Subtask）不做详细规划

4. 优化三——预定义优先：tool_selection 优先选预定义 Cypher（~200ms），命中则跳过 Text2Cypher

5. 优化四——缓存常见问题：Redis 缓存高频查询的 Cypher 结果

Result：预热后首次请求可降到 10-12 秒，缓存命中 <1 秒。

Q27：你如何控制 Token 成本？

Situation：每次请求调用 5-8 次 LLM，Token 消耗可观。

Task：在保证回复质量的前提下降低 Token 成本。

Action：

1. prompt 精简化——每个节点的 system prompt 保持 10-30 行，去掉不必要的解释性文字

Q28：Ollama Embedding 的 batch_size 为什么设成 2？

Situation：初始化时需要对 26 条查询做向量化。

Action：

2. 代价——26 条查询需要 13 次 HTTP 请求，初始化约 5 秒

3. 优化空间——如果 Ollama 升级或切换到云端 embedding API，可以增大 batch_size 或改成并行请求

Result：batch_size=2 下 Embedding 计算 100% 成功率。

Q32：如果 Neo4j 查询返回空结果，系统怎么处理？

Q35：幻觉检测本身会出错吗？

Situation：check_hallucinations 也是一个 LLM 调用，它也可能判断错误。

Task：控制幻觉检测的误判风险。

Action：

1. 误放行——数据库结果清晰时，check 的准确率较高；模糊时可能误判

2. 误拒绝——可能导致不必要的重试（最多 3 次），增加延迟但不影响最终结果

3. 设计取舍——宁可多一次重试（误拒绝），也不放行幻觉（误放行）。对客服场景，准确性优先于速度

4. 改进方向——加规则辅助（如检查回复中的产品名是否在数据库结果中出现过），减少纯 LLM 判断的不确定性

Result：当前设计偏保守（宁可误拒不可误放），约 85-90% 的检测准确率。

Q36：项目中遇到过什么印象深刻的 Bug？

Situation：开发过程中遇到了多个技术难点。

5. Bug 5（LangChain 模板解析错误）——Cypher 语句中的 {...} 被当作模板变量导致 INVALID_PROMPT_INPUT。修复：用 {{...}} 转义

4. 多步推理——"帮我挑一个性价比最高的智能音箱"需要先查产品、再查评价、最后综合推荐，ChatGPT 单次调用无法完成

Result：AI Agent = 通用 LLM + 私有数据 + 工具调用 + 安全护栏。

Q47：RAG（检索增强生成）和知识图谱查询有什么区别？

Situation：面试官可能追问基础知识。

Task：清晰解释两种技术路线的区别和适用场景。

Action：

1. RAG（向量检索）——文本切片 → embedding 向量化 → 存入向量数据库 → 用户查询时计算相似度 → 最相关的文本片段作为 LLM 的上下文。适合非结构化文本（文档、FAQ），擅长语义匹配但缺乏精确性

2. 知识图谱查询——结构化数据建模（实体+关系） → 用 Cypher/SPARQL 精确查询。适合有明确关系的数据（产品→类别、订单→客户），擅长精确查询和多跳关联

3. 本项目的混合方案——预定义 Cypher 用向量匹配（类似 RAG 的语义匹配思路）；Text2Cypher 动态生成精确查询（知识图谱优势）；FAQ 用向量匹配（类似 RAG）

4. 关键差异——RAG 返回"相似的文本段落"，知识图谱返回"精确的数据记录"。前者可能漏掉相关信息，后者需要精确的查询语句

Result：两种技术互补而非替代，本项目以知识图谱为主、向量匹配为辅。

Q48：Function Calling 在你的项目中用在哪里？

Situation：LangChain 的 bind_tools 本质是 OpenAI Function Calling 协议。

Task：说明 Function Calling 在项目中的具体使用。

Action：

1. Planner——llm.bind_tools([SubTask]) 让 LLM 输出结构化子任务列表

2. tool_selection——llm.bind_tools([predefined_cypher, cypher_query]) 让 LLM 选择最合适的查询工具

3. 原理——Function Calling = 告诉 LLM 有哪些"函数"（工具）可用 + 参数 Schema → LLM 返回应当调用哪个函数 + 参数 → 代码实际执行函数（查询 Neo4j）

4. 与传统 Function Calling 的区别——不是简单的"调用函数返回结果"，而是 LangGraph 的状态图 + Send() 并行分发

Result：Function Calling 是桥梁——LLM 负责"决定用什么工具"，LangGraph 负责"怎么执行、怎么组合结果"。

Q49：Prompt Engineering 在项目中有哪些应用？

Situation：项目中每个节点都有独立的 System Prompt。

Task：说明 Prompt 设计的核心思路。

Action：

1. 角色分离——每个节点一个 System Prompt（Router/Guardrails/Planner/Tool Selection/Summarize/Hallucination），互不干扰

2. 结构化输出引导——Router prompt 详细说明四种类型的边界，引导 LLM 准确分类

3. 约束注入——Guardrails prompt 注入业务范围描述 + 数据库标签，让 LLM 基于事实判断

4. 格式指定——Summarize prompt 指定回复风格（淘宝/京东客服语气），包含示例格式

5. 关键教训——Cypher 语句中的 {...} 与 LangChain 模板语法 {variable} 冲突，需要 {{ 转义

Result：良好的 Prompt 设计大幅提高了各节点的输出质量和格式一致性。

Q50：Embedding 模型 bge-m3 有什么特点？

Situation：面试官可能问为什么选这个 Embedding 模型。

Task：说明选型理由。

Action：

1. 多语言——bge-m3 支持中英文，电商场景以中文为主，中文 embedding 质量优于纯英文模型

2. 轻量——约 2GB，CPU 可运行（不需要 GPU），适合本地 Ollama 部署

3. 效果——在 MTEB 中文 benchmark 上表现优秀，1024 维向量，语义区分度高

4. 与其他模型对比——text-embedding-ada-002（OpenAI，需 API 调用，有费用）；m3e（中文模型，但效果略逊 bge-m3）；bge-m3 是性价比最优解

Result：bge-m3 在中文语义匹配效果和部署成本之间达到最佳平衡。

Q51：LangGraph 的 StateGraph 和 MessageGraph 有什么区别？

Situation：LangGraph 提供两种图类型。

Task：说明选择 StateGraph 的原因。

Action：

1. MessageGraph——专门为对话场景设计，状态自动管理 messages 列表，节点输入/输出都是消息。适合简单的对话 Agent

2. StateGraph——自定义状态 Schema（TypedDict），可以管理任意字段（router/tasks/cyphers/summary/answer/steps 等）。适合复杂的多步骤工作流

3. 选择 StateGraph——因为项目不仅需要管理 messages，还需要管理路由决策、子任务列表、查询结果、步骤追踪等多个维度的状态

4. InputState/OutputState——StateGraph 支持定义输入/输出状态子集，对外隐藏内部状态字段

Result：StateGraph 的灵活性是本项目复杂工作流管理的基础。
