# 灵犀智购 —— 智能客服系统项目解读报告

## 一、项目概述

**灵犀智购** 是一个基于 **LangGraph + Neo4j + DeepSeek** 构建的前后端分离智能客服系统，面向智能家居电商场景。用户登录后可浏览商品、通过 AI 客服助手查询产品信息、处理售后问题。核心 Agent 内部通过"路由 → 规划 → 工具选择 → 查询执行 → 结果汇总"的完整链路，自动从知识图谱检索数据并生成自然语言回复。

**一句话概括**：当用户在电商页面点击"咨询 AI 客服"提问时，后端 LangGraph Agent 会自动判断问题类型、拆解任务、从 Neo4j 知识图谱查数据、汇总成一段客服风格的自然语言回复，通过 SSE 流式推送到前端。

---

## 二、技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| **前端** | Vue 3 + Vite | 电商页面 + 客服聊天界面，编译后静态托管 |
| **后端框架** | FastAPI + Uvicorn | REST API + SSE 流式响应 |
| **Agent 框架** | LangGraph (StateGraph + 条件路由 + 嵌套子图) | Agent 流程编排 |
| **LLM 接入** | LangChain (ChatDeepSeek / Ollama) | 大模型调用 |
| **主 LLM** | DeepSeek `deepseek-chat` | 文字对话、Agent 推理 |
| **视觉模型** | 百炼 DashScope `qwen-vl-plus` | 用户上传图片分析 |
| **图数据库** | Neo4j Aura (云端) | 知识图谱存储与查询 (76+节点) |
| **关系数据库** | MySQL 8.0 + SQLAlchemy 2.0 (异步) | 用户/会话/消息持久化 |
| **向量匹配** | Ollama `bge-m3` + 余弦相似度 | 预定义 Cypher 查询的语义匹配 |
| **开发语言** | Python 3.12+ / JavaScript (ES Module) | |

│              数据层                               │
│  Neo4j (知识图谱)   MySQL (用户/会话)              │

**架构特点**：
- **前后端分离**：Vue 3 前端编译产物放在 `llm_backend/static/dist/` 下，由 FastAPI 统一托管
- **双 LLM 后端**：通过工厂模式 (`llm_factory.py`) 在 DeepSeek API 和 Ollama 本地模型之间灵活切换
- **嵌套 Agent 图**：父图负责路由分发，子图负责知识库查询的完整流程

## 五、核心业务流程（按执行顺序）

### 5.1 用户完整操作流程

```
1. 用户打开 http://localhost:8000
2. 前端加载 → 检查 localStorage 中是否有 token
   ├── 无 token → 显示登录页面
   └── 有 token → 调用 /api/validate-token 验证
       ├── 有效 → 显示商品页面
       └── 失效 → 清除 token，显示登录页面
3. 用户浏览商品（按分类筛选）
4. 用户点击"咨询 AI 客服"按钮 → 打开全屏聊天页
5. 用户输入问题（如"200元以内的智能灯泡有哪些？"）→ 回车发送
6. 前端构造 FormData，POST /api/langgraph/query
7. 后端 Agent 处理 → SSE 流式返回答案
8. 前端实时渲染（逐字显示）
9. 用户可继续追问（同一 conversation_id 保持对话上下文）
10. 关闭客服窗口 → conversationId 置空，下次打开是新会话
```

### 5.2 LangGraph Agent 内部执行流程（一请求的生命周期）

```
用户问题："200元以内的智能灯泡有哪些？"
    ↓
【节点1】analyze_and_route_query (路由判断)
    LLM 分析问题类型 → type: "graphrag-query"
    ↓
【条件路由】route_query → 进入 create_research_plan 分支
    ↓
【节点5】create_research_plan (知识库查询)
    ├── 连接 Neo4j
    ├── 构造子图输入 {messages, question, steps}
    ├── 调用子图 multi_tool_workflow.ainvoke()
    │   ↓
    │   【子图节点1】guardrails (安全门卫)
    │       判断"智能灯泡"在业务范围内 → next_action: "continue"
    │       ↓
    │   【子图节点2】planner (任务拆解)
    │       LLM 拆解问题 → 生成 1 个子任务: "查询200元以下照明产品"
    │       ↓
    │   【子图节点3】tool_selection (工具选择)
    │       LLM 分析任务 → 选择 predefined_cypher 工具
    │       参数: {query: "cheap_bulbs", parameters: {}}
    │       ↓
    │   【子图节点4】predefined_cypher (预定义查询执行)
    │       向量匹配 "200元以内灯泡" → 命中 cheap_bulbs (相似度 0.72)
    │       → 执行 Cypher: MATCH (p:Product)-[:BELONGS_TO]->(c:Category)
    │           WHERE c.CategoryName CONTAINS '照明'
    │           AND toFloat(p.UnitPrice) < 200
    │           RETURN p.ProductName, p.UnitPrice, p.UnitsInStock
    │       → Neo4j 返回 3 条记录
    │       ↓
    │   【子图节点5】summarize (结果汇总)
    │       LLM 将查询结果转为客服风格回复:
    │       "亲～目前200元以内的智能照明产品有3款：
    │        1. 华为智选智能灯泡E27 - 49.9元 (库存200)
    │        2. 小米米家LED智能灯带 - 79元 (库存150)
    │        3. 小度智能台灯Pro - 159元 (库存120)
    │        需要帮您详细对比吗？😊"
    │       ↓
    │   【子图节点6】check_hallucinations (幻觉检测)
    │       检查回复是否与查询结果一致 → 通过 → 结束子图
    │
    └── 子图返回 answer 字段
    ↓
返回 AIMessage → 追加到 graph 状态
    ↓
FastAPI process_stream()
    ├── 从 result["messages"][-1] 取最后一条消息
    ├── 手动分块（4字/chunk）
    └── SSE 流式推送到前端
```

### 5.3 四种路由分支的处理逻辑

| 路由类型 | 触发条件 | 处理节点 | 处理方式 |
|----------|----------|----------|----------|
| **general-query** | 闲聊、无关问题 | respond_to_general_query | LLM 直接回复，不查数据库 |
| **additional-query** | 意图模糊、缺少关键信息 | get_additional_info | Guardrails 先判断范围 → 在范围内则追问补充信息 |
| **graphrag-query** | 产品查询、订单、售后等 | create_research_plan | 进入完整的子图流程（6节点） |
| **image-query** | 用户上传了图片 | create_image_query | 视觉模型分析图片 → 返回描述 |

## 六、数据流转

### 6.1 请求数据流

```
前端 ChatPage
    │ FormData { query, user_id, conversation_id?, image? }
    ▼
FastAPI /api/langgraph/query
    │ 解析参数 → 保存图片 → 构造 thread_config
    ▼
LangGraph Agent
    │ InputState(messages=query)
    │ graph.ainvoke(input, thread_config)
    ▼
父图节点 → 子图节点 → Neo4j 查询
    │
    ▼
结果汇总 → result["messages"]
    │
    ▼
process_stream()
    │ 提取 messages[-1].content
    │ 4字/chunk SSE 流式输出
    ▼
前端 useChat.js sendMessage()
    │ ReadableStream 逐块读取
    │ 更新 aiMsg.content → Vue 响应式渲染
```

### 6.2 状态数据流 (LangGraph State)

```
InputState (用户输入)
    messages: [HumanMessage("200元以内的灯泡")]
    ↓
AgentState (完整状态)
    messages: [HumanMessage, AIMessage(路由结果), ..., AIMessage(最终回复)]
    router: {type: "graphrag-query", logic: "..."}
    question: "200元以内的灯泡"
    steps: ["analyze_and_route_query", "create_research_plan"]
    answer: "亲～目前200元以内的..."
    ↓ (通过 MemorySaver 持久化)
    ↓ (相同 thread_id 下次请求自动加载历史)
下一次请求 → InputState(messages="帮我挑一个性价比最高的")
    ↓
AgentState (自动追加历史)
    messages: [上一次对话..., HumanMessage("帮我挑一个...")]
```

### 6.3 子图状态数据流

```
InputState → {question, messages, steps}
    ↓
OverallState (子图全局)
    question: "200元以内的灯泡"
    messages: [对话历史]
    tasks: [Task("查询200元以下照明产品")]
    cyphers: [CypherOutputState(records=[3条产品])]
    summary: "亲～目前200元以内..."
    answer: "亲～目前200元以内..."
    hallucination_count: 0
    ↓
OutputState → {answer, question, steps, cyphers}
```

## 七、关键函数/类作用

### 7.1 状态机构建 (`lg_builder.py`)

| 函数 | 作用 |
|------|------|
| `analyze_and_route_query(state, config)` | **节点1**：LLM 分析问题类型，返回 `Router(type, logic)`。比喻：前台接待员判断来访者要去哪个部门 |
| `route_query(state)` | **条件路由函数**：根据 `state.router["type"]` 返回下一个节点名。比喻：路口的指路牌 |
| `respond_to_general_query(state, config)` | **节点2**：LLM 直接回答，不调任何工具。处理闲聊/通用问题 |
| `get_additional_info(state, config)` | **节点3**：Guardrails 过滤 + 要求用户补充信息。比喻：客服主管判断问题是否归我们管 |
| `create_research_plan(state, config)` | **节点5**：最复杂节点，内部调用子图 `multi_tool_workflow`。连接 Neo4j → 传入对话历史 → ainvoke 子图 → 提取 answer |
| `create_image_query(state, config)` | **节点4**：读取图片 → 压缩 → base64 → 调用视觉模型 API → 返回分析结果 |
| `graph` | 编译后的可执行 LangGraph 对象，被 `main.py` 导入使用。MemorySaver 提供多轮对话记忆 |

### 7.2 子图节点 (agentic_rag_agents/components/)

| 函数/类 | 作用 |
|----------|------|
| `create_guardrails_node(llm, graph)` | 创建 Guardrails 节点：用 LLM 判断问题是否与智能家居相关，不相关直接返回"抱歉" |
| `create_planner_node(llm)` | 创建 Planner 节点：LLM 将复杂问题拆解为 1-3 个独立子任务（SubTask） |
| `create_tool_selection_node(llm, tool_schemas)` | 创建 Tool Selection 节点：LLM 为每个子任务选择工具（predefined_cypher 或 cypher_query） |
| `create_predefined_cypher_node(graph, predefined_cypher_dict)` | 创建预定义查询节点：向量匹配 top-5 → 参数校验 → 逐个尝试执行 → 返回结果 |
| `create_cypher_query_node(llm, graph)` | 创建 Text2Cypher 节点：LLM 动态生成 Cypher → EXPLAIN 语法校验 → 执行（失败重试1次） |
| `create_summarization_node(llm)` | 创建汇总节点：Cypher 查询结果 → 客服风格自然语言。无数据时回退到对话上下文推理 |
| `create_check_hallucinations_node(llm)` | 创建幻觉检测节点：检查回复是否基于查询结果，无依据则重新生成（最多3次） |

### 7.3 路由边 (agentic_rag_agents/workflows/multi_agent/edges.py)

| 函数 | 作用 |
|------|------|
| `guardrails_conditional_edge(state)` | Guardrails 后路由：continue → planner，end → 直接结束 |
| `map_reduce_planner_to_tool_selection(state)` | **并行分发**：Planner 生成的 N 个子任务，通过 `Send()` 并行分发到 N 个 tool_selection 实例 |
| `summarize_conditional_edge(state)` | Summarize 后路由：有 cypher 结果 → 进入幻觉检测，无结果 → 直接结束 |
| `hallucination_conditional_edge(state)` | 幻觉检测后路由：通过 → 结束，不通过 → 回到 summarize 重新生成 |

### 7.4 LLM 服务层 (`services/`)

| 类 | 作用 |
|------|------|
| `LLMFactory` | **工厂类**：根据 `.env` 中的 `CHAT_SERVICE` / `REASON_SERVICE` 配置，创建 DeepSeek 或 Ollama 实例 |
| `DeepseekService` | DeepSeek API 封装：用 OpenAI SDK 兼容接口，支持流式 (`generate_stream`) 和非流式 (`generate`) 两种模式 |
| `OllamaService` | Ollama 本地模型封装：通过 aiohttp 调 Ollama REST API，支持流式输出 |
| `ConversationService` | 会话管理：创建会话、保存消息、获取历史、自动标题生成、增删改查 |

| 类/函数 | 作用 |
|----------|------|
| `VectorQueryMatcher` | 核心类：用 Ollama bge-m3 计算 26 条预定义查询的向量 → 用户问题向量 → 余弦相似度匹配 → 返回 top-K |

## 十一、关键设计决策与亮点

### 11.1 流式输出架构

**决策**：放弃 `graph.astream(stream_mode="messages")`，改用 `graph.ainvoke()` + 手动分块。

**原因**：`stream_mode="messages"` 会无差别捕获所有嵌套子图的 LLM 输出（包括英文推理过程、Cypher 语句、JSON 状态），严重污染前端显示。改用 `ainvoke()` 获取最终状态后手动分块，子图内部消息全部隔离在 ainvoke 内部。

### 11.2 预定义查询 + 向量匹配优先

**决策**：优先使用预定义 Cypher 查询（26条）+ 向量匹配，Text2Cypher 动态生成仅作为备选。

**原因**：LLM 动态生成的 Cypher 往往与数据 Schema 不完全匹配，查询准确率仅约 40%。预定义查询 + 向量匹配可覆盖 90%+ 场景，准确率大幅提升。

### 11.3 参数感知的降级匹配

**决策**：向量匹配 top-5 结果逐个尝试，跳过参数不足的候选，执行失败也自动降级到下一条。

**原因**：语义匹配只看向量相似度，不看参数兼容性。如"性价比最高的"和"按价格筛选产品"向量上很接近，但后者需要 `$price` 参数。加入参数校验和降级尝试机制后，空回复率从高频降为 0%。

### 11.4 对话上下文回退

**决策**：当所有 Cypher 查询都无结果时，summarize 节点回退到对话历史上下文进行推理回答。

**原因**：追问类问题（"帮我挑一个性价比最高的"、"第一个产品怎么样"）引用了上一轮的对话内容，无法被任何数据查询覆盖。回退到对话上下文让 LLM 从历史中推理，解决追问无法回答的问题。

### 11.5 双 LLM 后端工厂模式

**决策**：用工厂模式分别管理对话、推理、Agent 三个场景的 LLM 后端选择。

**原因**：对话用 DeepSeek（快），推理用 Ollama 本地模型（省钱），Agent 用 DeepSeek（稳）。通过 `.env` 一行配置即可切换，无需改代码。

def _embed_texts(self, texts):

### 🟢 性能类

#### 13.12 向量匹配每次初始化都重新计算

**现状**：`VectorQueryMatcher.__init__()` 每次都调用 `_compute_query_vectors()` 把所有预定义查询重新 embed 一遍。如果预定义查询不改，每次启动都是重复计算。

**建议**：加一个向量缓存——首次计算后把向量矩阵和查询名列表序列化到本地文件（如 `query_vectors.npz`），后续启动如果预定义查询没变就直接加载。

#### 13.13 Neo4j 连接每次查询都重新创建

**现状**：[lg_builder.py](llm_backend/app/lg_agent/lg_builder.py) 的 `create_research_plan` 和 `get_additional_info` 两个节点各调用一次 `get_neo4j_graph()`，每次请求都创建新的 `Neo4jGraph` 连接。

**建议**：在模块级别创建单例连接，或用 `functools.lru_cache` 缓存：

```python
from functools import lru_cache

@lru_cache(maxsize=1)
def get_neo4j_graph() -> Neo4jGraph:
    ...
```

#### 13.14 子图节点间数据传递冗余

**现状**：summarize 节点收到的 `state["messages"]` 包含完整对话历史，但实际只需要最后 6 条。当前代码虽已做了 `msgs[-6:]` 截断，但状态中仍存着全部历史在节点间传递。

**建议**：在阶段定义中考虑为 summarize 单独设置一个 `recent_context` 字段，避免大对象在子图节点间全量拷贝。
