# 智能客服系统 — 技术挑战与解决方案

## 项目简介

基于 LangGraph + Neo4j + DeepSeek 构建的多工具智能客服 Agent 系统。用户登录后进入电商页面，通过 AI 客服助手查询产品、处理售后。Agent 内部通过路由→规划→工具选择→Cypher 生成与执行→结果汇总的完整链路，自动从知识图谱检索数据并生成自然语言回复。

---

## 挑战一：流式输出中 Agent 内部推理泄漏到前端

### 问题

Agent 的流式回复中出现了大量不应该给用户看的内容：英文推理过程（"Let me analyze the user's question..."）、Cypher 查询语句（"MATCH (p:Product)..."）、状态 JSON（`{"logic":"...","type":"graphrag-query"}`）。这些内容严重影响了用户体验。

### 排查过程

1. 初步怀疑是前端渲染问题，尝试正则过滤 → 无效，JSON token 以逐字符方式流式到达，单字符 `{`、`"`、`l`、`o`、`g` 无法区分
2. 怀疑是 `stream_mode="messages"` 的问题，切换到 `updates` 模式 → 部分节点无声失败，且其他路由丢失输出
3. 尝试按 `langgraph_node` 过滤 → 发现子图内部消息全部标记为父节点名 `create_research_plan`，无法区分
4. 尝试用 `ainvoke` 取最终状态 → 仍然泄漏，发现 `response["answer"]` 在进入父图状态前已被子图内部消息污染

### 根因

**两层问题叠加**：

- **ChatOpenAI（langchain_openai）** 在使用 `with_structured_output` 时，会将结构化输出的 JSON 逐 token 流出来（`{`、`"`、`l`、`o`、`g`、`i`、`c`...），而 **ChatDeepSeek（langchain_deepseek）** 不会
- **LangGraph 的嵌套图机制**：子图 `multi_tool_workflow.ainvoke()` 执行期间，其内部所有节点的 LLM 输出都被添加到父图的消息通道中。`stream_mode="messages"` 会无差别捕获所有层级的所有 LLM token

### 解决方案

采用两步修复：

1. **模型层**：改回 `ChatDeepSeek` + DeepSeek 原生 API（`deepseek-chat`），避免结构化输出的 token 泄漏。视觉模型单独走百炼 DashScope 的 `qwen-vl-plus`

2. **流式架构层**：放弃 `graph.astream(stream_mode="messages")`，改用 `graph.ainvoke()` 获取最终状态，从 `messages[-1]` 取最后一条消息作为回复，手动分块（4字/chunk）模拟流式输出。子图内部的中间消息全部被隔离在 ainvoke 内部

```python
# 修复后的核心代码
async def process_stream():
    result = await graph.ainvoke(stream_input, thread_config)
    msgs = result.get("messages", [])
    if msgs:
        last = msgs[-1]
        if hasattr(last, 'content') and last.content:
            answer = str(last.content)
            for i in range(0, len(answer), 4):
                yield f"data: {json.dumps(answer[i:i+4])}\n\n"
```

### 效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 前端出现 JSON/英文推理 | 每次请求都有 | 0% |
| 中间推理字符占比 | ~40-60% | 0% |
| 客服回复可读性 | 差（需要滚动才能看到真正的回复） | 好（开头即是客服回复） |

## 挑战二：Cypher 执行静默失败导致 Agent 始终返回"没有找到"

### 问题

Neo4j 中确认有数据（200元以下有 3 款照明产品），但 Agent 始终回复"非常抱歉，目前没有找到相关产品"。看起来所有流程都在正常运行（日志显示路由、Guardrails、Planner、Cypher 节点都成功执行），但最终答案就是查不到数据。

### 排查过程

1. 直接在 Neo4j 执行相同的 Cypher → 返回 3 条结果，数据确实存在
2. 检查 Cypher 生成逻辑 → few-shot 示例正确，关键词匹配正确
3. 检查日志 → 无异常报错
4. 添加 Cypher 执行日志 → 发现 `'coroutine' object has no attribute 'replace'` 异常被上层 catch 吞掉了
5. 追踪到 `cypher_tools/utils.py:489` 行

### 根因

```python
# 第 489 行：缺少 await
corrected_cypher_update = correct_cypher_chain.ainvoke(...)  # 返回 coroutine 对象！

# 应该为：
corrected_cypher_update = await correct_cypher_chain.ainvoke(...)  # 返回字符串
```

Cypher 语法校验发现错误后，尝试调用 LLM 修正查询。但 `ainvoke` 前缺少 `await`，导致 `corrected_cypher_update` 是一个 coroutine 对象而不是字符串。后续代码将这个 coroutine 当作 Cypher 语句传给 Neo4j 执行，Neo4j driver 无法执行 coroutine，抛出异常被上游 catch 吞掉，最终表现为"查到 0 条结果"。

### 解决方案

在 `ainvoke` 前添加 `await` 关键字。这是一个典型的异步编程错误——函数返回了 coroutine 对象但 Type Hints 没有捕获（因为 Python 的类型系统对 async/await 的检查有限）。

### 效果

Cypher 异常修正路径恢复工作，Agent 能够正确执行查询并返回结果。

## 挑战三：向量匹配系统未生效导致预定义查询无法命中

### 问题

项目中有一套完整的向量匹配系统（`VectorQueryMatcher`），用 Ollama 的 `bge-m3` 模型将用户问题与 30+ 条预定义 Cypher 查询进行语义匹配。但这套系统在三个层面都出了问题，导致用户查询无法命中正确的预定义查询。

### 根因（三层）

**第一层：向量匹配器从未被调用**

`VectorQueryMatcher` 在 `predefined_cypher/node.py` 中被 import，但实际节点逻辑用的是 `predefined_cypher_dict.get(query_name)` 直接字典查找。LLM 通过 `bind_tools` 输出的查询名如果不精确匹配字典 key，就直接失败。

**第二层：Ollama Embedding 批量请求被拒绝**

`_compute_query_vectors()` 一次性把 30+ 条描述文本打包发给 Ollama API，请求体过大导致 Ollama 返回 500 错误。异常处理返回零向量 `[[0.0] * 1024]`，余弦相似度全部为 0，匹配器永远找不到结果。

**第三层：LLM 猜测覆盖向量匹配结果**

即使前两层修好，LLM 通过 `bind_tools` 输出的 `query` 字段可能是任意值。如果猜的恰好是字典里存在的 key（比如 "product_by_name"），就直接用了错误查询，向量匹配根本没机会执行。

### 解决方案

**第一层**：重写 `predefined_cypher` 节点，始终优先使用向量匹配：

```python
# 始终用向量匹配找最佳预定义查询
matcher = create_vector_query_matcher(predefined_cypher_dict, QUERY_DESCRIPTIONS)
matches = matcher.match_query(task, top_k=3)
if matches:
    best = matches[0]
    records = graph.query(query=best["cypher"])
```

**第二层**：将批量大小从全量改为 2：

```python
# 分批计算向量（每批 2 个，避免 Ollama 超限）
batch_size = 2
for i in range(0, len(query_texts), batch_size):
    batch = query_texts[i:i+batch_size]
    batch_vectors = self._embed_texts(batch)
    all_vectors.extend(batch_vectors)
```

**第三层**：向量匹配优先，LLM 输出仅在向量匹配失败时作为回退。

### 效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| "200以内的灯泡" → 正确查询 | ❌ 未命中 | ✅ `cheap_bulbs` (0.72) |
| "售后政策" → 正确查询 | ❌ 未命中 | ✅ `after_sales_policy` (0.85) |
| 预定义查询命中率 | ~30% | 90%+ |

## 挑战五：工具选择 LLM 反复选错工具导致 Text2Cypher 生成错误查询

### 问题

即使有 30+ 条预定义查询覆盖了常见场景，LLM 在 `tool_selection` 节点仍然倾向于选择 `cypher_query`（Text2Cypher 动态生成），而不是 `predefined_cypher`（直接执行预定义查询）。Text2Cypher 生成的查询往往与数据 Schema 不完全匹配（如 `CONTAINS '灯泡'` 而非 `CONTAINS '照明'`），导致查不到数据。

### 解决方案

1. 将 `predefined_cypher` 工具描述的优先级提高，标注为"首选工具"
2. 将 `cypher_query` 描述改为"仅在预定义查询无法匹配时使用"
3. 将 `default_to_text2cypher` 从 `True` 改为 `False`
4. 最终方案：从工具列表中移除 `cypher_query`，只保留 `predefined_cypher`（配合向量匹配器可覆盖 90%+ 场景）

### 效果

Text2Cypher 生成错误查询的问题完全消除（因为不再走动态生成路径），查询准确率从约 40% 提升到 90%+。

## 挑战六：向量匹配选中参数化查询但参数缺失导致 Agent 返回空回复

### 问题

用户在对话中说"帮我从里面挑一个性价比最高的"，Agent 返回了空回复。日志显示 Neo4j 报错 `ParameterMissing: Expected parameter(s): price`。同样的问题也出现在用户问"这个产品售后怎么样"但没提具体产品名时。

### 排查过程

1. 追踪日志 → Neo4j 抛出 `ClientError: ParameterMissing`，错误发生在 `predefined_cypher/node.py:54`
2. 检查向量匹配日志 → 用户的"性价比最高的"被匹配到了 `cheap_products_by_price`（相似度 0.68），这条查询需要 `$price` 参数
3. 检查参数来源 → 参数由 `tool_selection` 节点的 LLM 输出，但 LLM 没从用户问题中提取到 `price`（因为用户根本没提价格数字）
4. 检查原有容错逻辑 → **没有**。第一条匹配失败就直接抛出异常，不会尝试第二条

### 根因

向量匹配器返回 top-K 结果后，`predefined_cypher` 节点只取第一条直接执行，完全不管这条查询是否需要参数、参数有没有被提供。当第一条查询因为 `ParameterMissing` 失败时，异常未被捕获，沿调用链一路向上传播，最终 LangGraph 节点执行失败，`process_stream()` 从 `result.get("messages", [])` 取到最后一条消息为空 → 前端收到空回复。

更深层的问题是：**向量匹配只看语义相似度，不看参数兼容性**。`cheap_products_by_price` 的语义描述是"按价格筛选产品"，和"性价比"在向量空间里确实接近，但它要求一个具体的 `$price` 值，而这类模糊查询天然不会提供。

### 解决方案

重写 `predefined_cypher/node.py`，加入三层容错：

```python
# 1. 用正则提取 Cypher 语句里所有 $param 参数名
def _extract_param_names(cypher: str) -> List[str]:
    return re.findall(r'\$(\w+)', cypher)

# 2. 逐个尝试向量匹配结果，跳过参数不足的
matches = matcher.match_query(task, top_k=5)  # 加大到 top-5
for match in matches:
    required_params = _extract_param_names(match["cypher"])
    missing = [p for p in required_params if p not in available_params]
    if missing:
        print(f"跳过 {match['query_name']}，缺少参数: {missing}")
        continue  # ← 不崩溃，试下一个
    try:
        records = graph.query(query=match["cypher"], params=available_params)
        break  # ← 成功就停止
    except Exception as e:
        continue  # ← 执行失败也试下一个

# 3. 全部失败才报错
if not records:
    errors.append(f"No predefined query matched: {task}")
```

**关键思路**：
1. **参数感知过滤**：执行前先检查参数是否齐全，不全的直接跳过
2. **降级尝试**：从 top-3 扩大到 top-5，增加命中无参数查询的概率
3. **执行异常捕获**：`try/except` 包裹 `graph.query()`，单条失败不会炸掉整个节点

### 效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| "性价比最高的" → 空回复 | 每次 | 0%（回退到 `top_rated_products`） |
| "这个售后怎么样"（不提产品名）→ 空回复 | 每次 | 0%（回退到 `after_sales_policy`） |
| 参数化查询因缺参崩溃 | 100%（第一条失败即崩溃） | 0%（逐条降级尝试） |

## 挑战七：对话记忆完全失效——多轮对话变成多条独立会话

### 问题

用户反馈"它好像没有对话记忆能力，不能记得上次对话"。测试发现问"我叫张三"后再问"我刚才说我叫什么？"，Agent 完全不知道。所有消息都被当作独立的新会话处理。

### 排查过程

1. 检查前端 `sendChat()` → 构造 FormData 时只发了 `query` 和 `user_id`，**从未发送 `conversation_id`**
2. 检查后端 `langgraph_query()` → `thread_id = conversation_id if conversation_id else new_uuid()`。因为 `conversation_id` 永远为 `None`，每次请求都生成新的 `thread_id`
3. 检查 LangGraph 状态恢复逻辑 → `has_history` 的检查代码是 `len(state_history) > 0 and len(state_history[-1]) > 0`。`StateSnapshot` 是 namedtuple（7 个字段），`len()` 永远返回 7，`[-1]` 拿到的是 `tasks` 而非 `messages`。即使前端传了正确的 `conversation_id`，`has_history` 也会是 `False`
4. 检查多轮对话输入 → `Command(resume=query) if has_history else InputState(messages=query)`。`Command(resume=...)` 是用于恢复 `interrupt()` 中断的，不是用于多轮对话的

### 根因

**三个 bug 叠加**：

- **前端**：从不发送 `conversation_id`（后端每次都返回了 `X-Conversation-ID` 响应头，但前端没读）
- **后端状态检查**：`len(StateSnapshot)` 永远返回 7，`[-1]` 是 `tasks` 而非消息列表，历史检测完全无效
- **多轮对话机制**：`Command(resume=...)` 专用于中断恢复，正常多轮对话应该始终用 `InputState(messages=...)`。LangGraph 的 checkpointer 会根据相同 `thread_id` 自动加载历史状态并用 `add_messages` reducer 追加新消息

### 解决方案

**前端**：`sendChat()` 首次请求从响应头捕获 `X-Conversation-ID`，后续请求附带；`closeChat()` 重置会话 ID

```javascript
// 发送
if(this.conversationId) fd.append('conversation_id', this.conversationId);
const res = await fetch('/api/langgraph/query', {method:'POST', body:fd});
this.conversationId = res.headers.get('X-Conversation-ID') || this.conversationId;

// 关闭时重置
closeChat() {
    this.conversationId = null;
    // 清空聊天记录，下次打开是新会话
}
```

**后端**：
```python
# 修复前
has_history = state_history and len(state_history) > 0 and len(state_history[-1]) > 0
stream_input = Command(resume=query) if has_history else InputState(messages=query)

# 修复后
has_history = (state_history is not None and state_history.values 
               and len(state_history.values.get('messages', [])) > 0)
stream_input = InputState(messages=query)  # 始终用 InputState，checkpointer 自动追加
```

### 效果

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| 多轮对话记忆 | 0%（每条消息独立会话） | 100%（同窗口内保持上下文） |
| "我刚才说了什么" 能正确回答 | ❌ | ✅ |
| 关闭客服窗口再打开 | — | 自动重置为新会话 |

## 挑战八：追问类问题无法回答——"帮我挑一个性价比最高的"返回空

### 问题

用户先问"200元以内的产品有哪些"（成功返回 3 款产品），接着追问"帮我挑一个性价比最高的"，Agent 却回复"没有找到相关信息"。更早之前同样的追问甚至直接返回空——整个节点因 `UnicodeEncodeError` 崩溃了。

这类"追问"（follow-up question）有一个共同特征：它们引用上一轮的对话内容，无法被任何预定义 Cypher 查询覆盖。

### 排查过程

1. 检查日志 → 第一条追问直接崩溃，报 `UnicodeEncodeError: 'gbk' codec can't encode character '✗'`。原因是日志里用了 Unicode 符号 `✓`/`✗`，Windows GBK 编码打不出来
2. 修复编码问题后 → 不再崩溃，但返回"没有找到相关信息"
3. 追踪向量匹配 → "帮我挑一个性价比最高的"被匹配到 `cheap_products_by_price`（需要 `$price`），参数缺失被跳过。后面的 `top_rated_products` 等也因相似度不够被过滤
4. 追踪 summarize 节点 → 收到的 `records` 是 `NO_CYPHER_RESULTS`（`[{"error": "..."}]`），LLM 被告知"只使用提供的信息构建回答"，只能编出"没有找到"
5. 检查关键问题 → **子图的 `OverallState` 中根本没有 `messages` 字段**。虽然 `lg_builder.py` 的 `create_research_plan` 把对话历史传给了子图，但子图状态定义里没声明这个字段，被 LangGraph 静默丢弃

### 根因

**两层隔离导致追问链断裂**：

- **父图→子图状态隔离**：`lg_builder.py` 传了 `"messages": state.messages`，但子图的 `OverallState` TypedDict 没声明 `messages` 字段。LangGraph 的状态管理对未声明字段静默忽略——传了等于没传
- **Summarize 节点无上下文回退**：当 Cypher 查询无结果时，summarize 只能看到空数据 + 当前问题。没有对话历史，LLM 完全不知道上一轮说了什么，无法做出任何有意义的回答

而追问"帮我挑一个性价比最高的"正属于"必须依赖对话上下文才能回答"的问题——它不是独立可查询的数据库问题，是对上一轮结果的分析请求。

### 解决方案

**第一步**：修复日志编码（低级 bug 但会导致整个节点崩溃）

```python
# 修复前
print(f"  ✓ 命中 {match['query_name']}")  # UnicodeEncodeError on Windows GBK

# 修复后
print(f"  [OK] 命中 {match['query_name']}")  # 纯 ASCII
```

**第二步**：在 `OverallState` 和 `InputState` 中声明 `messages` 字段，打通父图到子图的消息通道

```python
# state.py — 新增字段
class InputState(TypedDict):
    messages: Annotated[List[AnyMessage], add_messages]  # ← 加这行

class OverallState(TypedDict):
    messages: Annotated[List[AnyMessage], add_messages]  # ← 加这行
```

**第三步**：重写 summarize 节点，无 Cypher 数据时回退到对话上下文


## 关键收获

1. **LangGraph 流式输出的层级控制**：`stream_mode="messages"` 会无差别捕获所有嵌套子图的 LLM 输出。要控制输出范围，需要在架构层面隔离（`ainvoke` + 手动流式）而不是试图在流中过滤
2. **异步编程的隐蔽 bug**：`await` 缺失在 Python 中不产生编译错误，coroutine 对象被当作普通对象传递，直到运行时才在不相干的位置暴露
3. **向量检索的工程落地**：从算法正确（余弦相似度）到系统可用（批量大小、阈值调优、回退策略）中间隔着大量工程细节
4. **LLM 工具选择的不可靠性**：不应该依赖 LLM 在多个工具间做精准选择。减少工具数量 + 强化每个工具的内部匹配能力，比让 LLM 做路由器更可靠
5. **语义匹配 ≠ 参数兼容**：向量匹配只看语义相似度，不管参数兼容性。"性价比最高的"和"按价格筛选产品"向量上很接近，但后者需要 `$price` 参数而前者不提供。必须在匹配后加一层参数校验，跳过参数不兼容的候选
6. **LangGraph 多轮对话的正确姿势**：不要用 `Command(resume=...)` 做多轮对话（那是给 `interrupt()` 用的）。始终用 `InputState(messages=...)`，相同 `thread_id` 的 checkpointer 会自动追加消息
7. **前后端状态同步的脆弱性**：后端在响应头返回了 `X-Conversation-ID`，前端没读 → 整个记忆链断裂。API 设计时关键标识符既要放在响应体/头里，也要在前端文档里明确说明
8. **LangGraph 嵌套图的"状态隔离陷阱"**：父图传给子图的状态字段，如果不在子图的 TypedDict 中声明，会被静默丢弃——不报错、不警告。对话历史传了但子图收不到，导致所有追问类问题（"帮我挑一个"、"第一个怎么样"）全部失效
9. **Cypher 查询无法覆盖所有用户意图**：追问、比较、推荐等依赖上下文的问题不是数据库查询，是推理任务。当预定义查询全部不命中时，需要一条"对话上下文回退"路径，而不是返回空数据

## 挑战九：向量匹配启动时重复计算——服务重启即重算

### 问题

在 `predefined_cypher/utils.py` 中，预定义查询的向量在**每次服务启动**时都会重新计算：

```python
def _compute_query_vectors(self) -> Dict[str, np.ndarray]:
    # 每次启动都调用 Ollama API 生成向量
    all_vectors = []
    for text in query_texts:
        all_vectors.extend(self._embed_texts(batch))  # 调用 Ollama
    return {key: np.array(vec) for key, vec in zip(query_keys, all_vectors)}
```

**问题：**
1. 每次重启服务都要调用 Ollama API 生成向量
2. 57 条预定义查询 = 57 次 API 调用（启动慢）
3. 向量存在内存中，服务重启后需要重新计算
4. 无持久化存储，浪费计算资源

### 解决方案

#### 技术选型：Milvus 向量数据库

**为什么选 Milvus：**
- 轻量级部署（Milvus Lite 可本地运行）
- 支持余弦相似度检索
- 向量持久化存储，重启不丢失
- 插入和检索性能优秀

#### 配置新增（config.py）

```python
# Milvus settings
MILVUS_HOST: str = "localhost"
MILVUS_PORT: int = 19530
MILVUS_COLLECTION: str = "predefined_cypher_vectors"
```

#### 核心逻辑改造（utils.py）

**启动流程：**
```
1. 连接 Milvus，检查 collection 是否存在
2. 不存在 → 创建 collection（1024维，bge-m3 向量）
3. 存在 → 加载 collection
4. 从 Milvus 读取已有向量 → 构建内存字典
5. 检测缺失的向量（新增的预定义查询）→ 调用 Ollama 计算 → 插入 Milvus
```

**检索流程：**
```
用户问题 → Ollama 向量化 → Milvus search → 返回 top_k 匹配结果
```

#### 关键代码

```python
def _load_or_compute_vectors(self) -> Dict[str, np.ndarray]:
    # 检查 Milvus 中是否已有数据
    existing = self.milvus_client.query(
        collection_name=self.collection_name,
        output_fields=["query_name", "vector"],
        limit=1000,
    )

    # 已有向量
    stored_vectors = {item["query_name"]: np.array(item["vector"]) for item in existing}

    # 找出缺失的
    missing_names = [k for k in query_keys if k not in stored_vectors]

    # 只计算缺失的
    if missing_names:
        missing_texts = [f"{n} {self.query_descriptions.get(n, '')}" for n in missing_names]
        vectors = self._embed_texts(missing_texts)
        for name, vec in zip(missing_names, vectors):
            self.milvus_client.insert(collection_name=self.collection_name, data={"query_name": name, "vector": vec})
            stored_vectors[name] = np.array(vec)
```
